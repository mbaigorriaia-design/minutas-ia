# Contexto del Proyecto: Generador de Minutas IA

Este proyecto consiste en una aplicación de Python (Streamlit) integrada con flujos de trabajo en n8n para la generación automatizada de minutas de reunión a partir de documentos en Google Drive.

## Arquitectura Actual
- **Frontend**: Streamlit (Python) corriendo en Docker (`frontend_ui`).
- **Backend Automático**: n8n corriendo en Docker (`n8n`).
- **IA**: Groq (LLMa 3.x) configurada en n8n mediante LangChain.
- **Entorno**: Docker Compose gestionando ambos servicios.

## Componentes Clave
- `frontend_ui/app.py`: Interfaz de usuario para pegar URLs de Drive y descargar minutas.
- `n8n_data/`: Almacenamiento persistente de configuraciones y base de datos de n8n.
- `docs/`: Documentación técnica detallada (Instalación, Integración, Estabilidad).

* **[2026-03-28]**: Estabilización definitiva del Pipeline RAG Local (Ollama + PostgreSQL). Se identificó y resolvió el problema de "contaminación de memoria" donde el LLM alucinaba participantes de reuniones anteriores. La solución arquitectónica consistió en: 1) Configurar un nombre de tabla dinámico (`={{'reunion_' + $execution.id}}`) en los nodos de PGVector para aislamiento absoluto por cada ejecución. 2) Reemplazar el nodo genérico `Basic LLM Chain` por el especializado `Question and Answer Chain` (`chainRetrievalQa`), asegurando que los documentos recuperados de Postgres sean inyectados correctamente en el prompt de Llama 3.2. 3) Aumentar el context window (`numCtx: 8192`) y el límite de recuperación (`topK: 15`) para procesar transcripciones extensas sin amnesia temporal.
* **[2026-03-29]**: Identificación de límites físicos de hardware (VRAM/RAM) en ejecución local con Ollama. Se descubrió que aumentar manualmente el `Context Length` (ej: a 32768 tokens) o utilizar el método Chunking `Map Reduce` (que lanza peticiones concurrentes en `Promise.all`), provoca un colapso total en la memoria de la máquina host (manifestado como `fetch failed` instantáneo o a los 5 minutos). Se documentó como regla estricta de arquitectura el uso de parámetros de contexto bajos (<=8192) y la obligatoriedad de usar el método secuencial `Refine` para evitar asfixiar el entorno local.
* **[2026-03-30]**: Identificación del **Timeout de 5 minutos inamovible de LangChain** en n8n. Se confirmó que todos los nodos violetas de IA (LangChain: `Summarization Chain`, `Basic LLM Chain`, etc.) tienen un límite interno rígido de 300 segundos por petición que no puede ser configurado desde la UI de n8n. Esto provoca `fetch failed` en cualquier transcripción cuyo procesamiento exceda los 5 minutos, incluso en modo `Refine`, debido a la lentitud del hardware local (CPU-only, sin GPU dedicada).
* **[2026-03-30]**: Creación del flujo experimental **"Minutas VENG Rapidas (Bypass Timeout)"** que elimina completamente los nodos de LangChain y utiliza un nodo `HTTP Request` nativo apuntando directamente a la API de Ollama (`http://ollama:11434/api/generate`) con un timeout de 40 minutos (2,400,000 ms). Este flujo logró procesar exitosamente una transcripción de 14 páginas en **36 minutos** sin errores de conexión, probando la viabilidad del bypass. Sin embargo, la calidad del output fue inferior (LLaMA se "mareó" con 32K tokens de contexto, olvidó los delimitadores `---MINUTA---`/`---JSON---` y no completó el JSON estructurado).
* **[2026-03-30]**: Intento de flujo híbrido **"Chunking+ByPass"** en n8n usando nodo `SplitInBatches` (loop secuencial) con `HTTP Request`. El nodo de loop no funcionó correctamente (saltó directamente a la salida "done" sin procesar ningún chunk). Se intentó entonces un `Code Node` todo-en-uno que hiciera las llamadas HTTP internamente, pero el **sandbox de seguridad de n8n** bloquea tanto `$helpers.httpRequest()` como `fetch()` dentro de los nodos de código, haciendo imposible esta arquitectura.
* **[2026-03-30]**: **Solución definitiva: Chunking+ByPass desde Python (Streamlit → Ollama directo)**. Se reescribió `app.py` para que el modo "Extenso 🟡" ejecute todo el pipeline de chunking secuencial directamente desde Python, sin pasar por n8n en absoluto. Cuando el usuario selecciona Extenso, Streamlit: 1) Corta el texto en fragmentos de 6000 caracteres con 300 de overlap. 2) Llama a Ollama directamente vía HTTP (`http://ollama:11434/api/generate`) con `num_ctx: 8192` y timeout de 30 minutos por chunk. 3) Acumula los resultados usando la técnica Refine (cada chunk refina la minuta previa). 4) Muestra una **barra de progreso en tiempo real** ("🧠 Procesando fragmento 2 de 7..."). 5) Parsea la minuta + JSON final y los presenta en la UI con botón de descarga. Esta arquitectura elimina simultáneamente: el timeout de LangChain, las restricciones del sandbox de n8n, y la sobrecarga de memoria del contexto masivo (32K). El modo Rápido (verde) sigue usando n8n normalmente para transcripciones cortas.
* **[2026-03-30]**: **Ajuste de timeout por chunk de 10 a 30 minutos**. Se descubrió que en hardware CPU-only, cada fragmento de 6000 caracteres puede tardar **hasta 20 minutos** en ser procesado por LLaMA 3.2 (3B). El timeout de 10 minutos por chunk era insuficiente y provocaba abortos prematuros del procesamiento. Se actualizó a 1800 segundos (30 min) por chunk.
* **[2026-03-30]**: **Benchmarks reales de rendimiento en hardware local (CPU-only)**:
  - Modo Rápido (reuniones cortas, <8 pág): ~3-5 minutos vía n8n + LangChain.
  - Modo Bypass single-shot (32K contexto, 14 pág): ~36 minutos (calidad inferior, LLaMA pierde formato).
  - Modo Extenso Chunking+ByPass (4 chunks de 6000 chars): ~15-20 min por chunk, total estimado **60-80 minutos** para 14 páginas.
  - **Oportunidad de mejora**: Usar `llama3.2:1b` (1 Billón de parámetros) en vez de `llama3.2:3b` podría reducir tiempos ~3x (de 20 min/chunk a ~7 min/chunk).

## Arquitectura Final Consolidada (2026-04-20)
- **Modo Rápido 🟢**: `Streamlit → n8n (/minutas-fast) → Ollama → Streamlit`. Para respuestas inmediatas.
- **Modo Extenso 🟡**: `Streamlit → n8n (/minutas-chunking) → Loop Secuencial (Chunks de 100k) → Ollama → Streamlit`. Diseñado para resistir archivos de 10MB+ y hardware limitado.
- **Gestión de Memoria**: Liberación automática de RAM cada 60 segundos.
- **Limpieza de Datos**: Filtrado de "ruido" y avatares integrado en el flujo antes de llegar a la IA.
- **Despliegue**: GitHub ➔ Portainer Stack (con Pull and Redeploy).
- **Timeouts**: 2 horas de espera máxima para evitar errores de conexión en el frontend.
